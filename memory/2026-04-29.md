## WSL → Windows GitHub 推送解决方案

### 问题
- WSL 中 git push 到 GitHub 一直失败（connect failed to port 443）
- 代理配置不生效

### 解决方案（用户建议）
1. 将 WSL 项目文件复制到 Windows 临时目录
2. 在 Windows 中用 git 推送
3. 清理临时文件

### 操作步骤
```powershell
# 1. 复制文件到 Windows
wsl cp -r /home/administrator/projects/secure-vault-sol/* /mnt/c/temp-secure-vault/

# 2. Windows git 推送
cd C:\temp-secure-vault
git config http.proxy http://127.0.0.1:7890
git push origin master

# 3. 清理临时目录
cmd /c "rd /s /q C:\temp-secure-vault"
```

### 结果
- ✅ secure-vault-sol 项目成功推送到 https://github.com/529349029/secure-vault-sol
- ✅ 临时目录已清理

### 我的回答
Clash Meta 默认只监听 127.0.0.1，WSL 访问不了。需要两步：

2. **WSL 中设置代理**：获取 Windows 局域网 IP（如 192.168.1.100），WSL 里 `export https_proxy="http://192.168.1.100:7890"`

## Clash Meta 代理共享给 WSL（续）

### 问题排查过程
1. Windows 防火墙三个配置文件（Domain/Private/Public）全部已关闭 ✅
2. Clash.Meta 进程（PID 14840）正在监听 `::`（IPv6全接口）和 `127.0.0.1` ✅
3. WSL 能 ping 通 192.168.3.20，但 telnet 192.168.3.20 7890 连接被拒绝 ❌

### 根因
Windows 防火墙规则 `proxy7890` 的 **EdgeTraversalPolicy 是 Block**（阻止边缘遍历），导致 WSL 穿过宿主机防火墙时被拦截。

### 修复方法
```powershell
Set-NetFirewallRule -DisplayName "proxy7890" -EdgeTraversalPolicy Allow
```

修复后 EdgeTraversalPolicy 变为 `Allow`，WSL → Windows:7890 的连接应该能通过。

### WSL 端设置
```bash
# 临时生效
export https_proxy="http://192.168.3.20:7890"
export http_proxy="http://192.168.3.20:7890"

# 测试
curl -x http://192.168.3.20:7890 https://www.google.com
```

### 永久生效
加到 `~/.bashrc` 或 `~/.zshrc`。

### 经验
- 即使防火墙全部关闭，Windows 仍有 EdgeTraversalPolicy 机制
- EdgeTraversalPolicy=Block 会阻止 LAN 外的设备穿越防火墙
- 以后配置 Clash Meta 共享时，记得检查 EdgeTraversalPolicy

## Clash Meta 代理共享给 WSL（续）- 根因确认

### 最终根因
**Clash Meta 进程只绑定了 127.0.0.1:7890，没有绑定 0.0.0.0**

排查结果：
- `netstat -ano | Select-String ":7890.*LISTEN"` 显示：
  - `TCP 127.0.0.1:7890 0.0.0.0:0 LISTENING 9476`（clash.meta-windows-386）
- config.yaml 里已有 `bind-address: "0.0.0.0"` 和 `allow-lan: true`，但**进程没有应用新配置**
- Windows 防火墙和 EdgeTraversalPolicy 问题之前已修，但根本问题是**进程需要重启**

### 解决方法
**重启 Clash Meta 进程**（任务管理器找到 clash.meta-windows-386 结束再启动，或用脚本重启）

重启后验证：
```powershell
netstat -ano | Select-String ":7890.*LISTEN"
# 应看到 0.0.0.0:7890 或 [::]:7890
```

然后 WSL 测试：
```bash
telnet 192.168.3.20 7890
curl -x http://192.168.3.20:7890 https://www.google.com
```

### 教训
- config.yaml 改了不等于进程用了 → 必须重启才能生效
- EdgeTraversalPolicy 之前已修复防火墙侧，但进程监听地址才是根本
- 以后排查网络问题时，先确认进程实际监听的地址
